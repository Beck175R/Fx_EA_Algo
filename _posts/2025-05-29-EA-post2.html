EA完成後の最適化手順ガイド

最適化結果で「稼働」か「再最適化」かを決める具体的フロー

1. はじめに

MT5でEA(Expert Advisor)が完成したら、いきなり実弾投入するのではなく最適化 → 評価 → 再最適化のループで"実戦に耐える設定"を探ります。本記事では、EA最適化の全体像と「稼働 or 再最適化」判断基準を詳解します。

2. 最適化の全体フロー

1. 検証環境を固定

テスターのモデル、ティックデータ、スプレッド、手数料を実運用相当で統一

2. パラメータ範囲を設計

"意味のある"上下限だけに絞り、解像度を決める

3. 初回最適化を実行

ブルートフォース/遺伝的アルゴリズムなど手法を選択

4. 結果をフィルタリング

過剰最適化を防ぐためOOS(Out-of-Sample)区間を必ず作る

5. 稼働可否を判定

この記事の核心部分

6. 再最適化 or 本番稼働

パラメータクラスターが安定しているかで行先を決定

7. 定期モニタリング

稼働後もKPIが基準を割れば再最適化

3. 評価指標を揃える

指標 目安 着目点
純益 (Net Profit) + (戦略により幅あり) 右肩上がりか
PF (Profit Factor) 1.5 以上 1.3 未満は要再考
最大 DD 資金の 20% 以下 DDが浅いほど安定
リカバリーファクター 2.0 以上 回復力の指標
Sharpe/Sortino > 1.0 が目安 ボラ調整済み利益
WFE (Walk-Forward Efficiency) 1.0〜1.5 1.0 未満は却下
パラメータ安定度 "谷間"ではなく"台地" クラスター幅を確認
ポイント: 1つでも基準割れがあれば「再最適化」が原則。

5. 「稼働 or 再最適化」判定ロジック

IF (PF ≥ 1.5 AND WFE ≥ 1.0 AND DD ≤ 20% AND ParamClusterStable): Deploy to Live (Small Lot) ➜ Monitor ELSE IF (ParamClusterUnstable OR WFE < 1.0): Redesign Parameter Grid ➜ Re-Optimize ELSE: Tighten Filters / Add Market Regime Check ➜ Re-Optimize

5-1. 「稼働」してよいケース

  • 指標がすべて基準を満たす
  • パラメータが"台地状"（近接値でも成績が大きく変わらない）
  • IS・OOS両区間とも右肩上がり
  • シャープレシオ低下が緩やか

実運用手順

  1. デモ → 0.1 lotなど極小ロットで1〜2週間走行
  2. モニタリングKPIを毎日自動集計
  3. DDが設計値を超えたら即停止し、再最適化へ

5-2. 「再最適化」すべきケース

症状 具体策
WFE < 1.0 OOS区間で過剰最適化 → IS期間を短縮・パラメータ分解能を粗くする
パラメータが鋭いピーク 安定度不足 → 範囲を広げ台地を探す
OOSでPF < 1.3 マーケットレジームが変化 → ボラティリティ/スプレッドフィルタ追加
DD > 許容 SL/TP・ポジションサイズを再設計
リアルでスリッページ悪化 テスト環境の手数料・遅延を実測値に更新

6. 再最適化フローを効率化するコツ

  • Walk-Forward Optimizer を自動化
    • Python で MT5 API を呼び、IS/OOS をスライド
  • パラメータクラスタリング可視化
    • パラメータ対 PF ヒートマップ例 (pandas + matplotlib)
  • 定期ジョブ化
    • 月次自動最適化でパフォーマンス維持
  • 性能低下アラート
    • 純益が 2σ 下振れで Telegram 通知

7. 運用後のモニタリング指標

  • ライブ PF と ウォークフォワード PF の乖離
  • 実DD/理論DD の比率
  • 約定速度・スリッページの推移
  • キャッシュバック/手数料収支が戦略に与える影響

8. まとめ

  1. 基準を数値で決めることで「感覚的な稼働」は排除
  2. パラメータの"台地"が安定運用のカギ
  3. IS/OOS/ライブの3段階で常にKPIを監視
  4. 自動化×定期再最適化で戦略寿命を延ばす

これでEA完成後の最適化から実運用まで、迷わず判断できるはずです。成功を祈ります！
